# ADR-0006: 视频录制与存储策略

**状态:** 已确定  
**日期:** 2026-06-01

## 背景

宠物主要求全程视频记录，作为行程透明度的重要组成部分。实时视频流在 MVP 阶段不实现，采用本地录制 + 事后上传方案。

## 决策

**本地录制 + 分段上传 + 7天保留**

- **录制粒度**：每 15 分钟生成一个视频片段，录制完成后立即尝试上传
- **上传失败处理**：本地缓存未上传片段，设备在线后自动补传，或行程结束后统一补传
- **保留期限**：视频片段在云端保留 7 天，宠物主可在期限内回放和下载
- **上传通道**：独立于传感器数据通道（视频体积大，不与 5 分钟传感器数据混合上传）

## 快照

- 周期性快照（每 N 分钟一张）作为补充，优先于视频让宠物主快速确认宠物状态
- 快照独立于视频录制，独立上传

## 理由

- **15 分钟段**：平衡存储和视频完整性，15 分钟足够记录完整的行程片段，且单段体积可控
- **立即上传 + 补传**：利用行程中持续的网络连接，行程结束后可能设备已离线或有剩余片段未传完
- **7 天保留**：宠物主行程结束后通常在一周内查看回放，超出后删除节省存储成本
- **独立上传通道**：视频上传耗流量大，与传感器数据分离便于网络策略控制和上传重试逻辑

## 不包含在 MVP 的功能

- 实时视频流（实时观看，P2 实现）
- 视频事件触发上传（异常事件立即上传，关键视频优先补传，P2 考量）
- 视频云端永久存储

## 影响

- 设备端需要足够的本地存储缓存未上传的视频片段（估算：15分钟视频约XX MB，待设备规格确认）
- 设备固件需要实现视频片段管理（录制、上传状态、补传队列）
- 云端需要视频转码（如设备端编码格式不统一）和存储服务
