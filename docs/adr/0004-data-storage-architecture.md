# ADR-0004: 传感器数据存储架构

**状态:** 已确定  
**日期:** 2026-06-01

## 背景

设备端 5 分钟周期上报大量传感器时序数据，视频独立分段上传。需要确定数据存储组合，在成本、运维复杂度和功能之间取得平衡。

## 决策

**存储组合：TimescaleDB + PostgreSQL + Redis + S3/OSS**

| 数据类型 | 存储方案 | 说明 |
|---------|---------|------|
| 传感器时序数据（GPS、温感、加速度、音频摘要、光感） | TimescaleDB | PostgreSQL 扩展，时序+关系一体，运维最简 |
| 业务数据（用户、驿站、行程、告警记录） | PostgreSQL | 成熟稳定，与 TimescaleDB 共用同一 PostgreSQL 实例 |
| 实时状态（在途箱子当前状态） | Redis | 热数据缓存，后台看板读取快，数据量小 |
| 视频片段 + 快照图片 | S3/OSS | 对象存储，低成本，7天保留 |
| 旅程报告（行程结束后聚合生成） | PostgreSQL JSON列 | 静态报告，不常修改，按需查询 |

## 理由

- **TimescaleDB**：内嵌 PostgreSQL，一套数据库搞定时序 + 关系数据，无需维护多套系统。TimescaleDB 的自动分区对时序数据查询性能友好。
- **Redis 精简**：只存放实时热数据（在途箱子的当前 GPS、温感、告警状态），容量小，运维成本低。
- **S3/OSS**：视频和快照存储成本低，无需复杂的文件服务器架构。
- **未来扩展**：时序数据积累后可挖掘镖师评估、路线优化等用途，TimescaleDB 支持 downsampling，不影响历史数据可用性。

## 影响

- 视频片段 7 天后自动删除（OSS 生命周期策略）
- 传感器原始数据保留期限：建议 30 天（时序数据量大，30 天后价值递减），具体待定
- 后台看板通过 Redis 缓存展示实时状态，历史数据从 TimescaleDB 查询
